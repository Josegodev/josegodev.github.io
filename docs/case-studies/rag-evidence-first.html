Caso tecnico

RAG documental evidence-first

Recuperacion documental con estados explicitos EVIDENCE_FOUND / NO_EVIDENCE y respuestas condicionadas por evidencia.

Problema

Un LLM puede responder de forma convincente aunque la base documental no soporte la respuesta. En contextos tecnicos, esto reduce confianza y aumenta el riesgo operativo.

Decision arquitectonica

Priorizar retrieval, evidencia y estado de recuperacion antes de redactar. Si el estado es `NO_EVIDENCE`, la respuesta debe declarar la limitacion en lugar de completar con suposiciones.

Flujo simplificado

Consulta -> normalizacion -> busqueda documental -> seleccion de chunks -> estado EVIDENCE_FOUND / NO_EVIDENCE -> respuesta con evidencia o marcador de no-evidencia.

Ejemplo sintetico

{
  "query": "Que componentes intervienen en una ejecucion trazable?",
  "retrieval_status": "EVIDENCE_FOUND",
  "chunks": [
    {
      "chunk_id": 101,
      "source": "public-demo/runtime-overview.md",
      "score": 4
    }
  ],
  "published_corpus": "synthetic_demo_only"
}

Que demuestra

Criterio para construir IA documental con control de alucinaciones, trazabilidad de fuentes y comportamiento explicito ante informacion insuficiente.

El corpus real no se publica. Para portfolio se usan ejemplos sinteticos que muestran contrato y comportamiento, no contenido privado.

Stack

Python, FastAPI, JSON, embeddings, chunking documental, modelos locales, contratos de respuesta.

Estado actual

Experimental. Flujo local orientado a validacion de arquitectura, calidad de chunking y trazabilidad de recuperacion.

Limitaciones

  • La calidad depende de los documentos de entrada.
  • El chunking requiere ajuste por dominio.
  • No incluye evaluacion automatica extensa de precision de retrieval.
  • La demo publica no incluye el SQLite ni corpus privados originales.
