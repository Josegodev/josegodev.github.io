Madurez tecnica

Estado real del repo fuente

Lectura honesta para reclutadores: que demuestra el sistema, que sigue siendo experimental y que no debe venderse como produccion.

Que esta maduro

  • Separacion conceptual entre runtime, contratos, adapters de proveedor, servicios y trazabilidad.
  • Backend FastAPI con endpoints claros para salud, documentos y chat.
  • Integracion Telegram usada como canal operativo para probar el backend.
  • Trazas JSON/JSONL con `trace_id`, latencia, estado RAG, errores y metricas de tokens cuando estan disponibles.
  • Evals deterministas para detectar drift de respuestas, longitud, latencia y metricas operativas.

Que es experimental

  • RAG documental dependiente de corpus privado, SQLite local y calidad de chunking.
  • Laboratorios separados para validacion/fallback y pruebas de proveedores.
  • Observabilidad local basada en stdout JSON y archivos JSONL.
  • Convivencia de runtime, scripts y labs que requiere curacion antes de publicarse como demo abierta.

Que no debe venderse como produccion

  • No esta validado para alta concurrencia.
  • No tiene empaquetado reproducible completo en la raiz publica.
  • No tiene observabilidad centralizada, SLOs ni monitorizacion productiva.
  • No debe publicar corpus, logs, SQLite, rutas locales, identificadores Telegram ni trazas reales.
  • No debe afirmarse cloud productivo ni seguridad enterprise.

Proximos pasos tecnicos

Reproducibilidad

Añadir `requirements` raiz o `pyproject`, Dockerfile minimo y comandos de arranque verificables.

Configuracion

Parametrizar URLs locales y separar defaults de desarrollo de configuracion de demo publica.

Healthchecks

Ampliar `/health` para comprobar proveedor LLM, disponibilidad RAG y escritura de trazas.

Demo publica

Crear corpus RAG sintetico, limpiar baselines y publicar trazas anonimizadas generadas para demo.
