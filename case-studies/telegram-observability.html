Caso tecnico

Web + Telegram + observabilidad

Telegram como canal operativo secundario para probar el backend con trazabilidad por ejecucion, control de errores y metricas utiles.

Problema

Las integraciones IA suelen fallar cuando no existe visibilidad sobre origen de la solicitud, proveedor/modelo usado, latencia, tokens, errores y estado de retrieval.

Decision arquitectonica

Usar Telegram como canal de prueba operativa conectado a un backend comun, registrar trazas JSONL por ejecucion y mantener el canal separado de la logica de modelo.

Flujo simplificado

Mensaje Telegram -> parser/servicio bot -> API backend -> RAG/proveedor LLM -> respuesta -> trace_id + JSONL con metricas y estado.

Traza sintetica

{
  "trace_id": "trc_public_demo_ok_001",
  "source": "telegram",
  "provider": "ollama",
  "retrieval_status": "evidence_found",
  "tokens_input": 1210,
  "tokens_output": 286,
  "latency_ms": 1730,
  "status": "ok"
}

Que demuestra

Capacidad para integrar IA en canales reales sin perder control operativo, manteniendo trazabilidad y separacion entre interfaz, backend, retrieval y modelo.

No se publican `user_id`, `chat_id`, texto de usuarios, tokens ni logs reales.

Stack

FastAPI, Telegram Bot API, HTML/CSS, JSON logs, Docker, modelos locales segun disponibilidad.

Estado actual

Experimental. Flujo preparado para pruebas locales e iteracion de metricas antes de endurecer despliegue.

Limitaciones

  • No incluye gestion avanzada de colas.
  • Sin SLOs formales ni panel externo de observabilidad.
  • El coste se estima segun metrica disponible y puede ser cero en modelos locales.
  • Observabilidad local basada en JSON/JSONL, no centralizada.
