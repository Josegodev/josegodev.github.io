Caso tecnico

Runtime local IA controlado

Vista publica curada de un backend IA local basado en FastAPI, RAG opcional, adapters de proveedor y trazabilidad JSON.

Problema

Muchos prototipos IA mezclan prompt, modelo, herramientas, reglas de negocio y salida final en una unica capa. Esto dificulta reproducir errores, aplicar politicas y entender que ocurrio en cada ejecucion.

Decision arquitectonica

Diseñar un runtime backend donde FastAPI mantiene contratos, servicios y trazabilidad; el modelo se consume mediante adapters intercambiables y no decide acciones operativas.

Flujo simplificado

Request / Telegram / API -> FastAPI -> servicio bot/backend -> RAG opcional -> provider adapter -> respuesta -> trace JSON.

Vista curada

El sistema fuente contiene varios componentes y laboratorios. Este caso publica solo la arquitectura defendible: runtime local, contratos, adapters, RAG y trazabilidad. No se publican corpus privados, rutas locales, tokens, logs reales ni artefactos de ejecucion.

Que demuestra

Capacidad para convertir un prototipo IA en un backend inspeccionable, con puntos claros de control, metricas, errores analizables y proveedores LLM sustituibles.

Stack

Python, FastAPI, Pydantic, JSON logs, Telegram API, RAG SQLite, Ollama/OpenAI adapters y scripts de evaluacion local.

Estado actual

Experimental. Backend local avanzado y demo presentable con curacion. No se presenta como despliegue productivo.

Limitaciones

  • Sin validacion de alta concurrencia.
  • Persistencia de observabilidad basada en archivos JSON.
  • No sustituye un sistema de monitorizacion productivo.
  • Conviven runtime, labs y scripts; la vista publica debe separar esas fronteras.
